# Подсистема управления доступом и аутентификации

## 1. Общее описание
Данный модуль реализует архитектуру управления доступом на основе ролей (RBAC) с использованием гранулярных разрешений и механизмом двухтокенной аутентификации (JWT). Решение построено на базе FastAPI и SQLModel с использованием асинхронного драйвера PostgreSQL.

Основной принцип абстракции прав выстроен по цепочке: Пользователь -> Роль -> Разрешение. Система спроектирована так, чтобы исключить жесткое связывание прав с бизнес-логикой, предоставляя возможность динамического управления матрицей доступов через административный интерфейс.

## 2. Структура данных
Логика модуля опирается на шесть таблиц, разделенных на два домена: аутентификация и авторизация.

### Домен аутентификации
*   **User**: Основная таблица пользователей. Содержит учетные данные (email, хэш пароля) и флаг активности `is_active`. Мягкое удаление аккаунта реализуется путем переключения этого флага в `False`.
*   **RefreshToken**: Связь один-к-одному с пользователем. Хранит текущий активный токен обновления и срок его действия. При каждом обновлении сессии предыдущая запись перезаписывается.
*   **BlacklistedToken**: Таблица для отзыва JWT. При логауте уникальный идентификатор токена (`jti`) записывается сюда со сроком его истечения. Это позволяет корректно обрабатывать инвалидацию токенов до истечения их времени жизни без необходимости полного удаления сессий.

### Домен авторизации (RBAC)
*   **Role**: Справочник ролей (например, `admin`, `moderator`). Имя роли уникально.
*   **Permission**: Справочник гранулярных разрешений. Каждое разрешение состоит из трех компонентов: уникального имени (в формате `ресурс.действие`, например `posts.moderate`), целевого ресурса и конкретного действия.
*   **UserRole и RolePermission**: Связующие таблицы (Many-to-Many). 
    *   Особенность таблицы `UserRole` — наличие полей `assigned_at` и `assigned_by`. Это обеспечивает аудит истории назначения ролей, фиксируя, какой администратор и когда выдал доступ.

## 3. Механизм аутентификации
Система реализует паттерн stateless аутентификации с разделением токенов:
*   **Access Token** (JWT): Время жизни — 15 минут. Передается в заголовке `Authorization: Bearer`. Используется для проверки прав на каждом запросе.
*   **Refresh Token**: Время жизни — 30 дней. Передается исключительно в `HttpOnly`, `Secure`, `SameSite=Strict` куке. Используется только для бесшовного продления сессии доступа.

При выходе из системы (logout) выполняется два действия: физическое удаление записи `RefreshToken` из базы и добавление `jti` текущего Access Token в `BlacklistedToken`.

## 4. Логика авторизации и обработка статусов
Архитектура разрешений основана на формате `ресурс.действие`. Это позволяет отказаться от базового CRUD в пользу точечных действий (например, `posts.publish` или `system.backup`).

### Обработка неавторизованных запросов (Роль Guest)
В системе нет отдельного костыля для анонимов. Роль `guest` является полноценной записью в БД. 
Если запрос приходит без токена, зависимость авторизации воспринимает это как пользователя с ролью `guest` и проверяет наличие требуемого разрешения у этой роли. Это позволяет единообразно защищать публичные эндпоинты.

### Семантика HTTP-статусов
Зависимость проверки прав строго дифференцирует ошибки:
*   **401 Unauthorized**: Запрашиваемое действие недоступно для анонимного пользователя (у роли `guest` нет такого разрешения) либо предоставлен невалидный/просроченный токен.
*   **403 Forbidden**: Пользователь успешно аутентифицирован, но у набора назначенных ему ролей отсутствует требуемое разрешение на выполнение действия.

### Иерархия предустановленных ролей
При инициализации создается базовая матрица:
*   **admin**: Суперпользователь. При инициализации получает специальный wildcard-маркер `*`, который программно транслируется во все существующие в БД разрешения.
*   **moderator**: Управление контентом (публикация, модерация, удаление чужих постов/комментариев), доступ к аналитике. Не имеет прав на управление архитектурой системы.
*   **user**: Стандартный аутентифицированный пользователь. Чтение профилей, управление собственным контентом.
*   **guest**: Только чтение публичных данных (посты, комментарии).

## 5. Руководство по интеграции

### Защита эндпоинтов
Для ограничения доступа используется зависимость `require_permission`, куда передаются ресурс и действие.

Для эндпоинтов, доступных без авторизации (если это позволяет роль guest), тип переменной должен быть указан как `Optional[User]`:
```python
from typing import Optional
from fastapi import APIRouter, Depends
from ..security.deps import require_permission
from ..models import User

router = APIRouter()

@router.get("/posts")
async def get_posts(
    user: Optional[User] = Depends(require_permission("posts", "read"))
):
    return {"data": [...]}
```

Для эндпоинтов, требующих обязательной авторизации, тип указывается жестко как `User`:
```python
@router.post("/posts")
async def create_post(
    user: User = Depends(require_permission("posts", "create"))
):
    return {"author_id": user.id}
```

### Управление матрицей доступов (Admin API)
Все эндпоинты управления защищены разрешением `roles.manage` и расположены в роутере `/admin`. 

Для изменения прав используется паттерн полной синхронизации (замены). Клиент передает полный массив ID разрешений или ролей, которые должны быть назначены. Сервисный слой полностью очищает старые связи и записывает новые, что исключает рассинхронизацию состояний между клиентом и сервером.

Основные методы:
*   `GET /admin/permissions` — получение полного справочника разрешений (необходимо для построения UI с чекбоксами).
*   `PUT /admin/roles/{role_id}/permissions` — перезапись списка разрешений для конкретной роли.
*   `PUT /admin/users/{user_id}/roles` — перезапись списка ролей для пользователя.
*(В эндпоинте назначения ролей пользователю реализована серверная защита: администратор не может изменить собственные роли, что предотвращает случайную блокировку самого себя).*

## 6. Инициализация системы
Модуль включает в себя скрипты первичной инициализации (`init_rbac_data.py`, `init_users_with_roles.py`), которые запускаются при старте приложения (в lifespan-событии). 

Скрипты обладают идемпотентностью: они проверяют наличие ролей и разрешений в БД перед их созданием. При первом запуске в систему автоматически добавляются три тестовых пользователя:
*   `admin@example.com` (Роль: admin)
*   `moderator@example.com` (Роль: moderator)
*   `user@example.com` (Роль: user)
